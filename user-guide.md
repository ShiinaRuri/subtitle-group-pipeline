# 字幕组协作平台使用教程

如果只想快速上手，先读“核心概念”和自己岗位对应的小节。如果你是监制、组管理员或超级管理员，重点阅读“管理人员完整操作手册”和“任务创建规则与管理建议”。

## 核心概念

### 1. 系统角色和岗位角色是两套东西

系统角色决定“你能管理什么”。

岗位角色决定“你在某个项目、某一集、某个任务里做什么”。

常见混淆：

- `member` 是系统角色，表示普通成员。
- `release` 是岗位角色，表示发布岗位。
- 一个发布岗位成员的系统角色通常仍然是 `member`。
- 不要把“发布”“翻译”“压制”理解成系统权限。它们是项目工作流岗位。

### 2. 项目按分集推进

一个项目可以包含多集。进入项目后，正确理解方式是：

1. 先选项目。
2. 再选集。
3. 在这一集下面处理任务流水线。

文件、任务、评论、版本记录都尽量绑定到具体项目、具体集、具体任务。

### 3. 默认流水线顺序

系统默认把每集制作拆成下面的串行流水线：

```text
片源 source
  -> 时轴 timing
  -> 翻译 translation
  -> 后期 post_production
  -> 压制 encoding
  -> 发布 release
```

默认含义：

- 片源是第一步，没有前置任务。
- 时轴依赖片源。
- 翻译依赖时轴。
- 后期依赖翻译。
- 压制依赖后期。
- 发布依赖压制。

### 4. 开项不会自动创建一堆默认任务

项目创建后只建立项目、分集、成员、模板配置等基础结构。任务由监制或更高权限人员手动创建。

这样做的原因：

- 不同项目不一定每一集都需要完整六步。
- 有些集可能先做片源校验，有些集可能直接已有时轴。
- 任务过早创建会造成大量空任务、假进度和错误联动。

### 5. 文件不是简单覆盖

平台的文件管理接近 git 的版本思路：

- 每次上传都会产生记录。
- 同名文件不会自动覆盖旧文件。
- 如果要替换某个已有文件，必须在上传时显式选择“替换已有文件”。
- 同一个任务多次提交会形成多个版本。
- 默认下载或下游引用时，会优先使用最新审核通过版本；没有审核通过版本时才使用最新版本。

### 6. 翻译是特殊串行

翻译岗位支持多人竞争式认领时间片段，但执行逻辑不是无序并行。

当前设计口径：

- 多个译者可以认领不同时间段。
- 翻译任务需要排序 ID。
- 第一个翻译任务从时轴成品开始。
- 第二个翻译任务从第一个翻译任务审核通过的成品继续。
- 第三个翻译任务从第二个翻译任务审核通过的成品继续。
- 以此类推。
- 后期只接最后一个翻译任务，也就是排序 ID 最大且审核通过的翻译成果。

这能减少多人同时改同一份 ASS 文件造成的冲突。

## 权限和角色

### 系统角色总览

| 系统角色 | 中文名 | 主要权限 |
| --- | --- | --- |
| `super_admin` | 超级管理员 | 全局最高权限，系统初始化后的首个管理账号，管理品牌、存储、注册、通知、成员、模板、健康检查等 |
| `group_admin` | 组管理员 | 管理成员、标签、模板、公告、存储、通知渠道、健康检查等大部分后台能力 |
| `supervisor` | 监制 | 创建项目、管理自己负责的项目、创建和分配任务、审核任务、调整分集、管理项目公告、项目 Wiki 等 |
| `member` | 普通成员 | 查看自己参与的项目、申请技能标签、领取或处理符合条件的任务、上传交付物、评论和查看通知 |

### 岗位角色总览

| 岗位角色 | 中文名 | 主要交付 |
| --- | --- | --- |
| `source` | 片源 | 片源文件、片源网盘链接、片源时长、校验说明 |
| `timing` | 时轴 | 时轴字幕文件，通常是 ASS/SSA/SRT/VTT 等 |
| `translation` | 翻译 | 翻译稿、分段翻译字幕、翻译疑问 |
| `post_production` | 后期 | 特效字幕、样式、字体、后期工程包 |
| `encoding` | 压制 | 内封/内嵌成品视频、压制参数、日志、网盘链接 |
| `release` | 发布 | 最终发布包、种子、网盘链接、发布文案、校验信息 |
| `supervisor` | 监制 | 审核、返工、重置、删除、统筹、项目资料维护 |

普通成员可以：

- 登录系统。
- 查看工作台。
- 查看自己参与的项目。
- 查看开放认领任务的项目。

- 处理分配给自己的任务。
- 上传自己任务需要的文件或链接。
- 评论任务和文件版本。
- 修改自己的昵称、头像、密码。
- 发起 QQ 换绑。

监制可以：

- 创建项目。
- 基于模板开项。
- 填写项目 QQ 群。
- 配置项目成员。
- 调整项目集数。
- 创建任务。
- 分配任务。
- 发布开放认领任务。
- 审核任务提交。
- 驳回任务提交。
- 重置任务状态。
- 删除任务。
- 删除项目文件。
- 编辑项目设置。
- 编辑项目交付清单。
- 管理项目公告。
- 编辑项目 Wiki。
- 查看项目级工作量。
- 访问所有岗位操作面板。

组管理员可以：

- 管理成员账号。
- 创建、禁用、删除普通账号。
- 重置成员密码。
- 管理成员个人信息。
- 直接授予或重置成员技能标签。
- 手动通过注册验证。
- 管理注册策略和岗位标签。
- 管理项目模板。
- 管理全局公告。
- 管理存储后端。
- 管理通知渠道。
- 查看全局健康检查。
- 管理归档和保留策略。

超级管理员额外可以：

- 修改系统名称。
- 修改系统 logo。
- 管理最高级系统配置。
- 自身账号受保护，不能删除，也不能降低或更改自己的权限。

### 重要权限原则

1. 只有监制或更高权限可以新建项目。
2. 只有监制或更高权限可以创建项目任务。
3. 只有监制或更高权限可以审核通过或驳回任务。
4. 成员不能自己通过自己的任务。
5. 成员只能在分配给自己的翻译任务里认领翻译片段。
6. 项目监制及以上权限可以删除项目文件。
7. 超级管理员账号不能被删除。
8. 超级管理员不能降低或更改自己的权限。
9. 模板修改只影响之后创建的项目，不会追溯修改已有项目。

## 账号、注册、验证和个人资料

### 注册流程

注册入口在登录页。

普通注册需要填写：

- 用户名。
- 密码。
- 确认密码。
- QQ 号。
- 意向岗位标签，只有管理员已经配置过标签时才显示。

密码规则：

- 8 到 128 个字符。
- 至少包含 1 个英文字母。
- 至少包含 1 个数字。
- 不能包含空格或换行。

用户名和 QQ 号规则：

- 用户名不能和已有账号重复。
- QQ 号不能和已有账号重复。
- 用户名是登录用字段，昵称不是登录名。

### 注册策略

管理员可以在“系统设置 -> 注册和标签”中配置注册策略：

- 禁止注册：新用户无法自助注册。
- 开放注册：注册后直接激活。
- QQ 群验证：注册后账号进入待验证状态。

### QQ 群验证

如果启用 QQ 群验证：

1. 用户注册后页面会显示验证 QQ 群和命令。
2. 命令格式为：

```text
/verify <验证码>
```

3. 用户必须在配置的 QQ 群发送该命令。
4. QQ 机器人桥接器把消息转发给后端。
5. 后端确认群号、验证码和账号后激活账号。

待验证账号再次登录时，不会进入系统，而是继续显示验证说明。

管理员也可以在成员管理中手动通过验证，防止 QQ 机器人故障时注册被卡住。

### 登录

登录用“用户名 + 密码”。

注意：

- 登录框不使用昵称。
- 例如账号用户名是 `fabu`，昵称是“发布”，登录时应该输入 `fabu`。
- 用户名首尾空格会被自动去掉。
- 如果密码错误，前端会提示“用户名或密码错误”。

### 忘记密码

登录页有“忘记密码”。

支持两种验证方式：

- 邮件验证码。
- QQ 机器人命令。

QQ 命令格式：

```text
/resetpass <验证码>
```

流程：

1. 登录页点击“忘记密码”。
2. 输入用户名。
3. 系统生成验证码。
4. 如果账号绑定邮箱，会发送邮件。
5. 如果账号绑定 QQ，会尝试通过 QQ 发送提示。
6. 用户也可以用绑定 QQ 向机器人发送 `/resetpass <验证码>`。
7. 回到页面输入验证码和新密码。

### 个人资料

个人资料页可以修改：

- 昵称。
- 头像。
- 密码。
- QQ 换绑。

头像上传：

1. 选择图片。
2. 前端打开一比一裁剪工具。
3. 拖拽、缩放裁剪正方形。
4. 确认后上传到默认存储后端。
5. 旧头像会自动删除。

QQ 换绑：

1. 在个人设置页输入新 QQ。
2. 生成两条命令。
3. 用旧 QQ 发送：

```text
/rebindqq-old <验证码>
```

4. 用新 QQ 发送：

```text
/rebindqq-new <验证码>
```

5. 回到页面刷新资料。

注意：

- 旧 QQ 必须是当前账号已绑定的 QQ。
- 新 QQ 不能被其他账号占用。
- 两条命令有有效期。

## 工作台和任务广场

### 工作台首页

登录后默认进入工作台。

工作台显示：

- 欢迎语。
- 全局公告顶部栏。
- 我的任务看板。
- 最新动态。
- 项目公告。

公告显示规则：

- 如果有置顶公告，优先显示置顶公告。
- 否则显示最新公告。
- 如果还有更多公告，会提示还有多少条并跳转公告页。

### 我的任务看板

我的任务看板按状态分组展示分配给自己的任务：

- 待处理。
- 进行中。
- 待审核。
- 已超期。
- 已完成。

点击任务卡片会进入所属项目。

### 任务广场和项目列表

项目列表页包含：

- 招募中项目。
- 我参与的项目。
- 全部项目。

开放认领的任务会在项目卡片上标明正在开放认领的岗位。

## 管理人员完整操作手册

本节专门面向监制、组管理员和超级管理员。

管理人员的核心目标不是“把所有按钮都点一遍”，而是保证项目流水线稳定、任务边界清楚、文件版本可追溯、下游解锁准确。

### 管理人员的三条主线

监制或更高权限人员管理项目时，建议始终同时盯住三条线：

1. 流程线：这一集现在卡在哪个岗位，下一步是否具备启动条件。
2. 文件线：当前岗位使用的是哪个审核通过版本，下游是否拿到了正确版本。
3. 人员线：任务是否已经分配给正确的人，开放认领是否设置了标签限制和截止时间。

不要只看任务状态。任务显示“已提交”不代表可进入下游，必须等监制审核通过；任务显示“进行中”也不代表前置一定正确，异常重置、删除前序任务、翻译排序错误都可能让流程看起来在走，实际已经断链。

### 每集从 0 到发布的管理流程

一集的标准流转可以按下面的检查点推进：

1. 创建片源任务。
2. 片源提交文件或网盘链接，并填写本集实际时长。
3. 监制审核片源，确认视频可用、时长可信、链接可访问。
4. 创建时轴任务，要求时轴从已审核片源开始。
5. 监制审核时轴，确认 ASS/字幕结构和基础轴点可用于翻译。
6. 根据片长和人力创建翻译任务，只有翻译任务填写排序 ID。
7. 为翻译任务设置必要标签、负责人或开放认领条件。
8. 译者按排序 ID 串行工作，后一段只基于前一段审核通过成果继续。
9. 监制逐段审核翻译，发现冲突时进入字幕冲突处理，不强行合并。
10. 最后一段翻译通过后，创建或启动后期任务。
11. 后期下载最大排序 ID 的翻译通过版本，提交特效字幕、字体或工程包。
12. 监制审核后期，确认字体、样式、Aegisub section 和效果完整。
13. 创建压制任务，压制按项目模板中的内封/内嵌成品配置执行。
14. 监制审核压制，确认画面、音轨、字幕封装、码率、文件大小和日志。
15. 创建发布任务，发布下载压制通过版本，提交种子、网盘链接、发布文案等。
16. 监制审核发布，确认交付清单全部完成。
17. 全集完成后，项目进入归档或复盘。

每一个检查点都应该有明确结论：通过、驳回、重置、补充说明，或暂缓。不要用口头聊天替代任务审核记录。

### 每个岗位的管理验收重点

| 岗位 | 管理验收重点 | 通过后解锁 |
| --- | --- | --- |
| 片源 | 文件或链接可用、时长填写正确、分辨率和音轨信息清楚、坏帧或缺帧已说明 | 时轴 |
| 时轴 | 字幕结构完整、对白轴点可靠、OP/ED 和屏幕字处理清楚、无明显重叠 | 翻译 |
| 翻译 | 排序 ID 正确、只基于上一审核通过版本、术语一致、ASS section 未破坏 | 下一翻译段或后期 |
| 后期 | 特效和样式完整、字体齐全、Aegisub 私有 section 保留、最终字幕可用于压制 | 压制 |
| 压制 | 内封/内嵌要求符合模板、码率/分辨率/音轨正确、视频可播放、日志完整 | 发布 |
| 发布 | 文件命名、网盘/种子、发布说明、校验信息和交付清单完整 | 完成或归档 |

### 监制和管理员分工建议

监制负责项目内流转：

- 决定每集任务怎么拆。
- 分配成员或开放认领。
- 审核提交物。
- 处理重置、驳回、删除和下游联动。
- 维护项目公告、Wiki、术语表和交付清单。

组管理员负责全局资源：

- 维护成员账号。
- 授予技能标签。
- 配置注册策略。
- 维护项目模板。
- 配置通知渠道、存储和健康检查。

超级管理员负责底座：

- 系统名称和 logo。
- 初始化配置。
- 默认存储后端。
- 数据库连接和升级。
- 最高权限账号保护。

监制不应依赖超级管理员来处理日常任务流转；超级管理员也不应绕过项目监制直接修改某一集的审核结论，除非是在排障或接管项目。

### 管理人员日常顺序

推荐按下面顺序工作：

1. 先检查系统设置是否完整。
2. 准备岗位标签。
3. 准备项目模板。
4. 开项并填写项目 QQ 群。
5. 配置成员或开放招募。
6. 按分集创建片源任务。
7. 片源提交并记录本集时长。
8. 创建时轴任务。
9. 时轴通过后创建翻译任务。
10. 翻译任务按排序 ID 串行推进。
11. 翻译全部通过后创建后期任务。
12. 后期通过后创建压制任务。
13. 压制通过后创建发布任务。
14. 发布通过后检查交付清单。
15. 项目完成后归档。
16. 归档期满后按保留策略清理旧版本。

### 监制开项前检查清单

开项前建议确认：

- 是否有可用存储后端。
- 存储后端是否健康。
- S3 凭据是否通过参数校验。
- 项目模板是否符合本项目类型。
- 模板里各岗位是否启用正确。
- 模板中翻译每人最大认领时长是否合理。
- 模板中成品配置是否明确。
- 项目 QQ 群号是否已经创建。
- 关键成员是否已有账号。
- 关键成员是否已有技能标签。
- QQ 通知渠道是否可用。
- 邮件通知是否可用。

### 管理人员不要做的事

不要：

- 在开项时一次创建所有集、所有岗位的全部任务。
- 在片源时长未确认前创建大量翻译片段任务。
- 给非翻译任务填写翻译排序 ID。
- 删除前序任务后不检查下游任务。
- 未审核通过上游文件就强行启动下游。
- 用同名上传来试图覆盖文件。
- 让普通成员自己通过自己的任务。
- 把岗位角色当成系统角色。
- 修改模板后期待已有项目自动变化。
- 在发布任务已经交付后随意重置压制或更早任务。

### 管理人员该优先关注的页面

监制重点页面：

- 工作台。
- 项目列表。
- 项目详情。
- 项目成员。
- 项目 Wiki。
- 项目文件。
- 项目公告。
- 工作量。
- 系统设置中的模板。

组管理员重点页面：

- 成员管理。
- 技能标签和注册策略。
- 项目模板。
- 存储后端。
- 通知渠道。
- 全局公告。
- 全局健康检查。
- 数据保留。

超级管理员额外重点页面：

- 系统品牌。
- 默认存储后端。
- 初始化配置。
- 数据库连接。
- 全局健康检查。

## 项目创建和开项

### 谁可以创建项目

只有监制或更高权限可以创建项目。

普通成员不会看到新建项目按钮，也不能通过后端接口创建项目。

### 创建项目入口

入口通常在：

- 工作台右上角“新建项目”。
- 侧边栏“项目”页面中的新建入口。
- 顶部“新建”菜单。

### 创建项目需要填写的内容

创建项目分多步：

1. 基本信息。
2. 选择模板。
3. 选择存储后端。
4. 配置成员。
5. 确认创建。

基本信息包括：

- 项目名称。
- 项目 QQ 群号。
- 项目类型。
- 季数。
- 集数。
- 标签。
- 描述。

项目 QQ 群号非常重要：

- 项目相关需要 QQ @ 的通知会发到项目群。
- 如果没有填项目群，QQ 项目通知可能无法投递到正确群。
- 项目群通常应在开项前建好。

### 模板选择

模板决定：

- 默认岗位流程。
- 各岗位是否启用。
- 每个岗位的名额。
- 任务分配方式。
- 上传策略。
- 通知策略。
- ASS 处理策略。
- 成品配置。
- 交付清单。
- 发布任务类型。
- 翻译每人最大认领时长。

模板修改只影响新项目。

已有项目创建时已经复制模板配置，后续模板变化不会自动改变已有项目。

### 存储后端选择

每个项目必须绑定一个启用中的存储后端。

存储后端用于：

- 项目文件。
- 任务交付物。
- 文件版本。
- 头像。
- Logo。
- 预签名上传。

如果没有可用存储后端，不能正常开项或上传。

### 成员配置

成员配置按模板里的岗位槽位展示。

可以：

- 直接给某个岗位分配成员。
- 留空后续再分配。
- 开项后在项目成员页继续调整。

选择成员时应看昵称，必要时再核对用户名。

### 确认创建

最后一步会展示：

- 基本信息。
- 项目 QQ 群号。
- 模板。
- 存储后端。
- 已分配成员。

确认无误后再创建。

### 开项后第一件事

开项后不要急着创建后期、压制、发布任务。

建议先：

1. 打开项目。
2. 检查分集列表。
3. 检查项目成员。
4. 检查项目 QQ 群。
5. 检查交付清单。
6. 创建第一集片源任务。

## 分集和流水线

### 分集是什么

分集是项目内实际推进的单位。

每一集有自己的：

- 片源时长。
- 任务流水线。
- 文件集合。
- 评论上下文。
- 流水线进度。

### 调整集数

监制或更高权限可以调整项目总集数。

往多了调：

- 系统自动创建新集卡片。
- 补全必要数据库字段。
- 新集初始没有任务。

往少了调：

- 页面会要求选择要删除的集。
- 删除前会检查这些集是否有任务或文件。
- 如果不为空，会进行二次提示。
- 删除集会影响该集下的任务、文件和历史记录，应谨慎操作。

### 片源定义本集时长

每一集的时长由片源岗位提交时填写。

注意：

- 不以项目模板里的固定时长为准。
- 翻译片段认领依赖本集时长。
- 如果片源时长填写错误，后续翻译分段会受影响。

### 流水线进度条

项目每集会显示流水线进度条。

含义：

- 已完成的步骤会点亮。
- 下一步进行中时，两个节点之间的线条会显示部分填充。
- 进度条只是展示，不用于筛选任务。
- 不要把进度条当成按钮。

## 任务创建规则与管理建议

本节是监制和更高权限人员最重要的部分。

### 任务创建总原则

创建任务前先回答五个问题：

1. 这个任务属于哪一集。
2. 它属于哪个岗位。
3. 它需要基于哪个上一流程审核通过结果。
4. 谁可以做它，是否需要技能标签。
5. 做完后会解锁哪一个下游任务。

如果这五个问题答不清楚，先不要创建任务。否则后续最容易出现状态错乱、文件拿错、认领权限错误、下游提前启动等问题。

### 任务创建决策表

| 场景 | 推荐创建方式 | 说明 |
| --- | --- | --- |
| 每集第一步 | 创建片源任务 | 片源没有前置流程，可以直接作为起点 |
| 片源还没审核 | 不创建或只预建锁定的时轴任务 | 不允许时轴实际开始 |
| 时轴已通过 | 创建翻译任务 | 翻译任务要规划排序 ID 和片段范围 |
| 翻译任务很多 | 先建完整排序，再开放认领 | 避免中途插队导致排序错乱 |
| 某段翻译缺人 | 开放认领并设置标签 | 不要把无标签限制的任务暴露给所有人 |
| 翻译全部通过 | 创建后期任务 | 后期基于最大排序 ID 的翻译结果 |
| 后期已通过 | 创建压制任务 | 压制必须看到成品配置 |
| 压制已通过 | 创建发布任务 | 发布基于压制审核通过版本 |
| 只是补充材料 | 用监制补交或文件上传 | 不一定需要新建流水线任务 |
| 任务建错且未开始 | 删除 | 删除后检查下游是否为空 |
| 任务已参与流转 | 优先重置或驳回 | 保留历史更利于追溯 |

### 任务发布前检查

点击创建或发布任务前，监制建议检查：

- 分集是否选对。
- 岗位是否选对。
- 任务模板是否匹配岗位。
- 非翻译任务是否没有排序 ID。
- 翻译任务排序 ID 是否唯一且连续。
- 负责人是否是项目成员。
- 开放认领任务是否设置必要标签。
- 截止时间是否合理。
- 上传策略是否允许该岗位要交付的文件类型。
- 任务说明是否写明上一流程输入和交付物。
- 如果需要 QQ 通知，项目 QQ 群是否已填写。

这一步比后期修状态便宜得多。

### 谁能创建任务

只有监制或更高权限可以创建项目任务。

普通成员不能新建任务。

### 任务必须归属到具体集

新建任务时必须选择：

- 项目。
- 分集。
- 岗位。
- 任务名称。
- 任务说明。
- 分配方式。
- 负责人或开放认领条件。

不要创建没有明确分集归属的制作任务。

### 使用任务模板

新建任务按钮打开后，可以选择岗位任务模板。

每个岗位有常用模板，例如：

- 片源整理。
- 片源校验。
- 时轴制作。
- 时轴校对。
- 正片翻译。
- 翻译校对。
- 后期特效。
- 字体与样式打包。
- 成品压制。
- 压制质检。
- 发布打包。
- 发布投递。

选择模板后：

- 会快速填充任务标题和说明。
- 仍然可以手动修改。
- 模板只是创建任务的辅助，不等于项目模板。

任务模板的正确用法：

- 用它减少重复输入。
- 用它统一同一岗位的说明格式。
- 用它提醒需要提交哪些材料。
- 不要把它当成任务状态机开关。
- 不要用错误岗位模板创建任务后再手工改成别的岗位。

如果任务模板选错，建议直接改回正确岗位模板重新填写说明。尤其是翻译、压制、发布三类任务，说明里涉及上一流程文件、成品参数和发布资产，不能只改标题。

### 翻译排序 ID

只有翻译任务需要排序 ID。

规则：

- 第一个翻译任务通常为 `1`。
- 第二个翻译任务为 `2`。
- 依次递增。
- 同一集内不要重复。
- 非翻译任务不要显示或填写排序 ID。

排序 ID 的影响：

- 排序 ID 为 1 的翻译任务下载上一流程结果时，拿时轴结果。
- 排序 ID 为 2 的翻译任务下载上一流程结果时，拿排序 ID 为 1 的翻译审核通过结果。
- 排序 ID 为 3 的翻译任务下载上一流程结果时，拿排序 ID 为 2 的翻译审核通过结果。
- 后期任务下载上一流程结果时，拿排序 ID 最大的翻译审核通过结果。

管理建议：

- 不要临时插入重复 ID。
- 如果必须插入中间段，先重新规划所有翻译任务排序。
- 调整排序前确认已有文件和审核状态。
- 翻译排序错了会造成认领、下载上一流程结果、后期解锁全都混乱。

### 手动指派和开放认领

任务通常有两种分配方式。

手动指派：

- 监制指定负责人。
- 适合片源、时轴、后期、压制、发布等责任边界清楚的岗位。
- 适合重要集、紧急任务、关键成员。

开放认领：

- 满足条件的成员自行领取。
- 适合翻译分段、简单校对、招募任务。
- 可以设置需要哪些技能标签才允许领取。

建议：

- 关键链路任务优先手动指派。
- 招募阶段可以开放认领。
- 开放认领必须配置清楚需要的岗位标签。
- 没有标签限制的开放任务，所有成员都可能看到并尝试领取。

管理建议：

- 核心链路优先“明确负责人”，例如片源、时轴、后期、压制、发布。
- 需要招募时才开放认领，例如翻译分段或临时校对。
- 开放认领不要替代项目成员管理。一个成员长期参与项目，仍应把他加入项目成员列表。
- 对时间敏感的任务，开放认领也要设置截止时间和提醒。
- 认领规则越开放，审核负担越重。

### 标签资格限制

开放给用户认领的任务可以限制为“只有获得某些标签的用户可领取”。

例如：

- 翻译任务需要“翻译”标签。
- 时轴任务需要“时轴”标签。
- 压制任务需要“压制”标签。
- 发布任务需要“发布”标签。

如果成员没有对应标签：

- 不应让他领取受限任务。
- 可以让他先去技能档案申请标签。
- 管理员也可以直接授予标签。

### 前置任务为空时不启动下游

系统会保护流水线不被断链。

如果某个环节的前序环节没有任务：

- 下游任务不会自动启动。
- 下游任务应保持待发布或等待状态。
- 不会因为“找不到前序任务”就直接变成进行中。

例外：

- 片源是第一步，没有前置流程。
- 片源任务可以作为每集起点创建并开始。

管理建议：

- 不要删除上游任务后忘记补建。
- 如果确实跳过某个岗位，应在项目模板或项目规则中明确禁用该岗位，而不是直接删任务造成断链。
- 如果某集不需要某岗位，监制应人工确认后调整流程，不要留下含糊状态。

### 任务依赖怎么理解

任务依赖不是简单看岗位名称，而是看“这一集里哪一个审核通过结果能作为输入”。

常规依赖：

- 时轴依赖片源。
- 翻译排序 ID 为 1 的任务依赖时轴。
- 翻译排序 ID 大于 1 的任务依赖上一排序 ID 的翻译任务。
- 后期依赖最大排序 ID 的翻译任务。
- 压制依赖后期。
- 发布依赖压制。

如果同一岗位存在多个辅助任务，例如“时轴制作”和“时轴校对”，监制要明确哪个任务的审核通过结果才是正式下游输入。辅助任务可以存在，但不能让下游误用未确认版本。

### 审核通过后的联动预期

监制点击通过后，应该发生这些事：

- 当前任务进入审核通过或完成状态。
- 当前任务的审核通过文件版本成为下游默认输入。
- 如果下游任务已经存在且前置条件全部满足，下游可以进入可处理状态。
- 如果下游任务不存在，系统不会凭空创建，监制需要手动创建。
- 如果下游前序环节为空，下游不会自动启动。
- 如果这是最后一个翻译排序任务，通过后才允许后期进入下一步。

如果审核通过后下游没有变化，先检查前序任务是否完整、翻译排序是否连续、下游任务是否已创建、下游任务是否被冻结。

### 重置、驳回、删除怎么选

监制遇到问题时，优先按下面规则判断：

- 提交物小问题，需要原负责人返工：驳回。
- 提交物基于错误上游版本，需要重新做：重置。
- 任务建错且还没有人处理：删除。
- 上游重大变更导致下游都不能用：重置上游并让系统联动下游。
- 某岗位确认不需要：调整项目流程或模板，避免直接删掉前序造成断链。
- 发布产物已经外发：先公告和记录，再重置或重新发布。

驳回通常只影响当前任务。重置会影响下游。删除会破坏历史链路。管理上宁可多写一条审核意见，也不要用删除清理已经发生过的工作。

### 任务创建的推荐时机

推荐做法：

1. 创建片源任务。
2. 片源通过后创建时轴任务。
3. 时轴通过后创建翻译任务。
4. 翻译任务全部规划清楚后再开放认领。
5. 翻译全部通过后创建后期任务。
6. 后期通过后创建压制任务。
7. 压制通过后创建发布任务。

可以提前创建下游任务吗？

可以，但要理解：

- 下游会被前置条件锁住。
- 前置未完成时不能开始。
- 前序为空时不会启动。
- 提前创建太多任务会增加误操作风险。

### 每个岗位任务数量建议

片源：

- 每集至少 1 个片源整理任务。
- 复杂项目可增加片源校验任务。

时轴：

- 每集通常 1 个时轴制作任务。
- 如需双人检查，可增加时轴校对任务。

翻译：

- 短集可 1 个翻译任务。
- 长集可按时间段拆多个翻译任务。
- 每个翻译任务必须有排序 ID。
- 翻译任务数量不要超过实际可管理的串行段数。

后期：

- 每集通常 1 个后期特效任务。
- 字体整理可单独建任务。

压制：

- 每集至少 1 个成品压制任务。
- 如果压制和质检分人，可拆成压制任务和质检任务。

发布：

- 每集或每批 1 个发布打包任务。
- 如果发布渠道复杂，可拆成发布打包和发布投递。

### 任务说明怎么写

任务说明建议包括：

- 本任务要做什么。
- 需要下载哪个上一流程结果。
- 需要提交什么文件。
- 是否允许网盘链接。
- 是否有命名要求。
- 是否有特殊参数。
- 截止时间。
- 审核重点。

示例：

```text
请基于本集时轴通过版本完成 00:00:00-00:12:00 的翻译。
提交 ASS 文件。保留原有 section，不要删除 Aegisub 私有 section。
遇到术语不确定请在任务评论中 @监制。
```

### 任务截止时间

设置截止时间后：

- 超期任务会被后台任务标记为超期。
- 系统会触发通知升级。
- 超期不会自动回收任务。

监制应根据项目节奏手动协调超期任务。

## 任务状态和流转规则

### 状态表

| 状态 | 中文 | 含义 |
| --- | --- | --- |
| `pending_publish` | 待发布 | 任务已存在，但前置条件未满足，或等待监制发布 |
| `claimable` | 可领取 | 任务开放认领，符合条件的成员可以领取 |
| `assigned` | 已指派 | 任务已指定负责人，但尚未开始 |
| `in_progress` | 进行中 | 负责人正在处理 |
| `submitted` | 已提交 | 成员已提交结果，等待审核 |
| `review_approved` | 审核通过 | 监制已通过审核，可作为下游输入 |
| `review_rejected` | 已驳回 | 审核未通过，需要返工 |
| `completed` | 已完成 | 任务完成并进入最终完成态 |
| `overdue` | 超期 | 超过截止时间，需要关注 |
| `frozen` | 已冻结 | 上游取消或异常导致暂不可处理 |

### 常见流转

手动指派任务：

```text
pending_publish -> assigned -> in_progress -> submitted -> review_approved -> completed
```

开放认领任务：

```text
pending_publish -> claimable -> assigned -> in_progress -> submitted -> review_approved -> completed
```

驳回返工：

```text
submitted -> review_rejected -> in_progress -> submitted
```

成员退回：

```text
assigned/in_progress/review_rejected/overdue -> claimable
```

任务超期：

```text
assigned/in_progress/submitted 等 -> overdue
```

### 审核规则

需要审核的岗位：

- 翻译。
- 后期。
- 压制。
- 发布。

监制或更高权限可以：

- 通过。
- 驳回。
- 填写审核意见。
- 重置状态。
- 删除任务。

普通岗位成员不能：

- 通过自己的任务。
- 驳回自己的任务。
- 绕过审核推进下游。

### 重置任务

监制或更高权限可以重置任务。

重置用途：

- 已完成任务发现问题。
- 上游文件替换。
- 下游基于错误版本制作。
- 任务状态卡死。
- 需要重新提交。

重置影响：

- 当前任务回到可处理状态。
- 已开始、已提交或已完成的下游任务会被联动重置。
- 下游成员会收到通知。
- 非发布岗位保留历史文件版本。
- 发布岗位的旧发布产物会被丢弃。

重置前建议：

1. 看清当前任务状态。
2. 检查下游已有多少任务。
3. 检查下游是否已经上传文件。
4. 在任务评论或项目公告说明原因。
5. 重置后通知相关成员重新下载上一流程结果。

### 删除任务

监制或更高权限可以删除任务。

删除前必须考虑：

- 它是不是后续任务的前置。
- 它有没有文件。
- 它有没有评论。
- 它有没有审核记录。
- 它是否已经被下游引用。

建议：

- 未开始、误创建的任务可以删除。
- 已经参与流水线的任务优先重置或取消，不要轻易删除。
- 删除前序任务后，检查下游是否被锁住。
- 如果要减少流程环节，优先通过模板或项目设置禁用该岗位。

### 取消任务

取消任务会冻结尚未开始的下游任务。

已经进行中的下游任务不会被自动回收，但会产生提醒。

适用场景：

- 某一环节不再需要。
- 当前任务方向错误。
- 项目计划变化。

### 成员主动退回任务

成员可以把已领取或已指派任务退回。

适用场景：

- 时间不够。
- 认领错误。
- 无法处理当前任务。

退回会记录审计日志。

## 各岗位操作指南

### 片源岗位

片源是每集流水线的起点。

#### 片源需要做什么

片源岗位负责：

- 提供可用于制作的原始视频。
- 提交片源网盘链接或上传文件。
- 填写本集实际时长。
- 记录分辨率、音轨、字幕轨、是否有缺帧或坏帧。
- 给时轴说明片源版本。

#### 片源可以提交什么

允许：

- 视频文件。
- 原盘或压缩包。
- 工程包。
- 网盘链接。
- 校验说明。

常见格式：

- `.mkv`
- `.mp4`
- `.mov`
- `.ts`
- `.m2ts`
- `.avi`
- `.webm`
- `.zip`
- `.7z`
- `.rar`

#### 片源提交步骤

1. 进入项目。
2. 选择分集。
3. 打开片源任务。
4. 下载或查看任务说明。
5. 上传片源文件，或添加网盘链接。
6. 填写片源时长。
7. 补充校验说明。
8. 点击提交任务。
9. 等待监制审核。

#### 片源坑位

- 不填时长会影响翻译分段。
- 片源换版会影响所有下游。
- 如果片源替换，监制应重置下游任务。
- 网盘链接要确认有效期和提取码。
- 原盘或大体积文件建议走 S3 预签名分片直传。

### 时轴岗位

#### 时轴需要做什么

时轴负责基于片源制作时间轴。

包括：

- 对白时间点。
- OP/ED。
- 屏幕字轴点。
- 分行断句。
- 避免重叠。
- 保持 ASS 基础结构。

#### 时轴提交什么

常见提交：

- `.ass`
- `.ssa`
- `.srt`
- `.vtt`
- `.ttml`
- `.txt`

#### 时轴操作步骤

1. 打开时轴任务。
2. 点击快捷下载上一流程结果，获取片源。
3. 制作时轴。
4. 上传字幕文件。
5. 在评论中说明特殊轴点。
6. 提交任务。
7. 等待监制审核。

#### 时轴坑位

- 不要用未审核片源做正式时轴。
- 时轴改动会影响所有翻译任务。
- 如果时轴审核后又修改，监制要考虑级联重置翻译和后续任务。

### 翻译岗位

翻译是本系统最特殊的岗位。

#### 翻译任务必须有排序 ID

每个翻译任务都有一个排序 ID。

排序 ID 决定：

- 下载哪个上一流程结果。
- 串行推进顺序。
- 后期最终接哪个翻译成果。

#### 翻译认领

翻译支持分段认领。

认领前提：

- 任务分配给你，或你符合开放认领条件。
- 你拥有需要的翻译标签。
- 片段不和别人已认领片段重叠。
- 不超过项目设置的每人最大认领时长。

认领界面会显示整体认领情况。

#### 翻译串行规则

排序 ID 为 1：

- 基于时轴审核通过结果开始。

排序 ID 为 2：

- 等排序 ID 1 审核通过后开始。
- 基于 ID 1 的审核通过成果继续翻译。

排序 ID 为 3：

- 等排序 ID 2 审核通过后开始。
- 基于 ID 2 的审核通过成果继续翻译。

依此类推。

#### 翻译提交步骤

1. 打开翻译任务。
2. 查看任务排序 ID。
3. 查看分段认领情况。
4. 认领自己的时间段。
5. 下载上一流程审核通过结果。
6. 在本地翻译自己的片段。
7. 保留 ASS 结构。
8. 上传翻译稿。
9. 提交任务。
10. 等待监制审核。

#### 翻译文件要求

可以提交：

- `.ass`
- `.ssa`
- `.srt`
- `.vtt`
- `.ttml`
- `.txt`

ASS 注意：

- 不要删除 `[Script Info]`。
- 不要删除 `[V4+ Styles]`。
- 不要破坏 `[Events]` 格式。
- 尽量保留 Aegisub 私有 section。
- 不确定的术语写评论，不要随意改全局样式。

#### 翻译坑位

- 不要直接拿时轴结果做第二段、第三段翻译。
- 第二段以后必须基于上一排序 ID 的审核通过结果。
- 不要改自己片段以外的大量内容。
- 不要删除样式 section。
- 不要和别人认领重叠时间。
- 不要跳过前一个翻译任务强行提交。
- 如果排序 ID 错，后期拿到的结果会错。

### 后期岗位

#### 后期需要做什么

后期负责：

- 字幕样式。
- 特效。
- 屏幕字。
- 字体整理。
- 排版。
- 特殊效果。

后期的上一流程结果应该是最后一个翻译任务的审核通过结果，也就是排序 ID 最大的翻译结果。

#### 后期可提交什么

- 特效 ASS/SSA。
- 字体文件。
- 后期工程包。
- 说明文档。

常见格式：

- `.ass`
- `.ssa`
- `.ttf`
- `.otf`
- `.ttc`
- `.zip`
- `.7z`
- `.aep`
- `.psd`

#### 后期步骤

1. 打开后期任务。
2. 快捷下载上一流程结果。
3. 确认拿到的是最后一个翻译任务结果。
4. 制作特效、样式和字体包。
5. 上传后期字幕或工程包。
6. 提交审核。

#### 后期坑位

- 不要拿第一段翻译结果做后期。
- 不要遗漏字体。
- 不要破坏 ASS section。
- 如果翻译重置，后期必须重新核对输入版本。

### 压制岗位

#### 压制需要做什么

压制负责生成成品视频。

项目模板中会配置：

- 内封成品。
- 内嵌成品。
- 分辨率。
- 帧率。
- 编码器。
- 编码器预设。
- 码率。
- 成片大小。
- 音频编码方式。
- 音频码率。
- 音频声道数。
- 额外参数。

压制任务页面会展示这些成品要求。

#### 压制可以提交什么

- 视频成品。
- 压制日志。
- 压制工程包。
- 网盘链接。

常见格式：

- `.mkv`
- `.mp4`
- `.mov`
- `.webm`
- `.zip`
- `.7z`
- `.txt`
- `.json`

#### 压制步骤

1. 打开压制任务。
2. 下载后期审核通过结果。
3. 查看任务中的成品配置。
4. 按内封或内嵌要求压制。
5. 上传视频或添加网盘链接。
6. 上传日志或参数说明。
7. 提交审核。

#### 压制坑位

- 不要忽略内封和内嵌的区别。
- 不要用未审核后期文件压制。
- 大视频上传需要存储后端和浏览器能访问 S3 endpoint。
- 压制通过后如果上游重置，压制任务会被联动重置。

### 发布岗位

#### 发布需要做什么

发布负责最终投递。

包括：

- 整理最终发布包。
- 发布种子。
- 发布网盘链接。
- 记录校验信息。
- 编写发布文案。
- 确认渠道投递。

#### 发布可以提交什么

- 成品视频。
- 外挂字幕。
- 发布包。
- 种子。
- 网盘链接。
- NFO。
- Markdown 或 TXT 发布说明。

#### 发布步骤

1. 打开发布任务。
2. 下载压制审核通过结果。
3. 核对文件命名和版本。
4. 整理发布资产。
5. 上传种子、说明文件或发布包。
6. 添加网盘链接。
7. 提交审核。
8. 通过后项目可进入完成或归档准备。

#### 发布坑位

- 发布任务是最容易受上游重置影响的任务。
- 上游压制或更早流程被重置时，发布旧产物会被丢弃。
- 不要在压制未审核通过前发布。
- 网盘链接要确认有效期、提取码和访问权限。
- 发布后仍要保留发布记录，方便回溯。

## 文件管理、版本历史和在线预览

### 文件管理入口

文件管理页先展示项目卡片。

正确流程：

1. 进入“文件”页面。
2. 选择项目卡片。
3. 进入该项目文件视图。
4. 再查看文件、链接和版本历史。

不要在没有项目上下文时直接上传文件。

### 文件命名

上传后系统会按岗位和任务进行命名。

示例：

```text
翻译_12.ass
压制_成品压制.mkv
发布_发布打包.zip
```

这样做是为了让文件在项目文件筒中可追溯。

### 新建文件和替换文件

上传时有两个模式：

新建独立文件实体：

- 默认模式。
- 即使同名也不会合并。
- 适合不同人上传不同候选文件。

显式替换已有文件：

- 必须手动选择要替换的文件。
- 会形成该文件的新版本。
- 适合修订同一份交付物。

### 版本历史

每个文件可以有多个版本。

文件列表只显示一行，并通过历史记录按钮查看：

- 版本号。
- 上传者。
- 上传时间。
- 变更说明。
- 是否当前版本。
- 是否审核通过版本。
- 下载按钮。

### 在线预览

项目文件管理支持单击预览。

支持：

- ASS/SSA 等字幕文本预览。
- 多版本 diff。
- GitHub 风格红绿 `+/-` 差异。
- 视频在线播放。

### 多版本评论

任务评论可以关联：

- 文件版本。
- ASS 字幕行号。

这表示评论可以追溯到指定版本，而不是只评论“当前文件”。

### 文件删除

项目监制及以上权限可以直接删除文件。

删除前应确认：

- 文件是否是下游任务正在使用的审核通过版本。
- 是否有评论引用该版本。
- 是否是发布任务产物。
- 是否应先归档或保留。

### 下载链接

下载会生成临时链接。

规则：

- 同一用户同一文件会复用未过期链接。
- 即将过期时重新生成。
- 视频文件有更长的安全余量。
- TTL 可由管理员全局配置，项目可覆盖。

### 大文件上传

大文件上传使用 S3 预签名分片直传时：

- 浏览器直接传对象存储。
- 后端负责生成预签名 URL 和完成登记。
- 上传超时时间按大文件场景放长。

坑位：

- S3 CORS 必须允许浏览器 PUT/POST。
- endpoint 必须从浏览器可访问。
- bucket 权限和 key 权限必须正确。
- 项目必须绑定对应存储后端。

## 字幕合并、去重和冲突处理

### ASS 处理口径

系统会解析 ASS 文件结构。

会关注：

- Script Info。
- Styles。
- Events。
- Dialogue。
- Comment。
- 时间范围。
- 文本内容。
- Aegisub 特殊 section。

目标：

- 保留结构。
- 只合并字幕文本变更。
- 不强行覆盖样式和私有 section。

### 合并逻辑

翻译稿合并采用接近 git diff/patch 的思路。

能自动处理：

- 完全相同的重叠行。
- 非冲突文本差异。
- 可确定来源的顺序变更。

不能自动处理：

- 同一时间段不同译文。
- 结构冲突。
- 样式或 section 破坏。
- 多个版本都修改了同一行且无法判断取舍。

### 冲突处理

出现冲突时：

- 系统创建冲突记录。
- 不会强行合并。
- 通知监制或指定冲突处理人员。
- 在线去重页显示冲突。
- 处理后写回新合并版本。

### 在线去重页面

在线去重页面支持：

- 选择两个版本比较。
- 红绿 diff。
- 时间轴重叠可视化。
- 冲突条目查看。
- 保留或合并决策。
- 生成新的合并版本。

权限：

- 普通成员可查看自己相关内容。
- 监制或指定冲突处理人员可处理冲突。

## 项目公告、通知和 QQ 群提醒

### 公告类型

公告分为：

- 全局公告。
- 项目公告。

全局公告：

- 管理员创建。
- 所有用户可见。
- 工作台顶部只展示最新或置顶的一条。

项目公告：

- 项目监制或管理员创建。
- 项目成员可见。
- 用于项目内注意事项。

### 通知类型

系统会在这些场景发通知：

- 任务分配。
- 任务认领。
- 任务提交。
- 审核通过。
- 审核驳回。
- 下游解锁。
- 下游级联重置。
- 文件上传。
- 加入申请。
- 项目公告。
- @ 提及。
- 超期升级。
- 合并冲突。

### 通知渠道

支持：

- 站内通知。
- 邮件。
- QQ。

管理员在“系统设置 -> 通知渠道”中配置：

- SMTP 邮件服务。
- QQ 机器人桥接器地址。
- QQ bot secret。

通知渠道页面有测试按钮：

- 邮件测试需要填写测试邮箱。
- QQ 测试需要填写群号和目标 QQ 号。

### QQ 项目群 @

如果通知需要 QQ @，系统应该发到项目群。

项目群来自项目开项时填写的 QQ 群号。

坑位：

- 项目没填 QQ 群号，项目通知无法准确投递。
- 成员没绑定 QQ，无法精确 @。
- QQ 机器人不在群里，消息会发送失败。
- QQ 桥接器 secret 不一致，后端心跳或发送会失败。

## 项目 Wiki、术语表和制作规范

项目 Wiki 用于维护：

- 制作说明。
- 翻译规范。
- 术语表。
- 角色译名。
- 地名。
- 片中专有词。
- 样式约定。
- 发布说明。

支持：

- Markdown。
- 表格块。
- 审核流程。
- 监制审批。

如果启用 Wiki 审核：

- 普通成员提交修改后进入待审核。
- 监制或管理员查看 diff。
- 通过后才进入正式展示。

建议：

- 翻译开始前先建术语表。
- 后期开始前补齐样式规范。
- 发布前补齐发布命名规则。

## 工作量、归档、回收站和数据保留

### 工作量

工作量页面向：

- 个人。
- 项目监制。
- 管理员。

普通成员在工作台看自己的任务。

监制可以看项目成员任务分布。

管理员可以查看全局工作量。

### 归档

项目完成后建议归档。

归档后：

- 项目进入只读或复盘状态。
- 主要制作任务不再继续变更。
- 文件版本进入保留策略管理。
- 仍可查看历史。

### 删除和回收站

项目删除策略是：

```text
先归档 -> 软删除进入回收站 -> 超期物理清理
```

回收站支持：

- 查看已删除项目。
- 恢复项目。
- 彻底删除。

管理员可以配置保留天数。

### 数据保留

数据保留策略用于控制：

- 归档项目旧版本清理。
- 回收站物理删除。
- 下载临时链接清理。
- Wiki 审核策略。

坑位：

- 归档清理会保留最终通过版本，但可能清理中间版本。
- 回收站物理删除不可恢复。
- 删除前应确认是否需要备份。

## 系统设置

系统设置按子卡片集中管理。

### 注册和标签

用于：

- 开关注册。
- 配置 QQ 验证群。
- 创建岗位标签。
- 审核标签申请。
- 查看申请理由。

标签岗位类型要选对：

- 片源。
- 时轴。
- 翻译。
- 后期。
- 压制。
- 发布。
- 监制。

### 成员管理

管理员可以：

- 创建账号。
- 设置系统角色。
- 启用或禁用账号。
- 重置密码。
- 删除账号。
- 编辑头像、昵称、用户名、QQ。
- 手动通过验证。
- 授予标签。
- 重置标签状态。

注意：

- 删除账号是数据库删除。
- 超级管理员不能删除。
- 超级管理员不能降低自己的权限。
- 用户名和 QQ 不能重复。

### 系统品牌

超级管理员可以修改：

- 系统名称。
- Logo。
- 浏览器标题。
- 浏览器图标。

Logo 上传会打开一比一裁剪工具。

Logo 存在默认存储后端。

### 存储后端

管理员可以管理：

- 本地存储。
- S3。
- S3 兼容存储。

配置 S3 时会先做参数校验。

不正常的 secret、endpoint、bucket、权限配置不应保存。

### 项目模板

模板配置包括：

- 工作流岗位。
- 每个岗位启用与否。
- 岗位名额。
- 分配方式。
- 翻译最大认领时长。
- 必要标签。
- 上传策略。
- 通知策略。
- ASS 策略。
- 成品配置。
- 交付清单。
- 发布任务类型。

成品配置包括：

- 内封。
- 内嵌。
- 分辨率。
- 帧率。
- 编码器。
- 编码器预设。
- 码率。
- 成片大小。
- 音频编码方式。
- 音频码率。
- 音频声道数。
- 额外参数。

### 通知渠道

配置：

- 邮件 SMTP。
- QQ 桥接器。
- QQ bot secret。

建议保存后立刻测试。

### 全局健康检查

显示：

- 数据库连接状态。
- 数据库类型。
- 数据库版本。
- QQ 机器人桥接器配置状态。
- QQ 机器人在线状态。
- 最近心跳。

注意：

- 没配置就显示未配置。
- 配置了但没心跳就显示未连接。
- 不应显示假在线。

## 常见坑位和排查

### 登录后回到登录页

检查：

- 用户名是否填的是用户名，不是昵称。
- 密码是否正确。
- 账号是否 active。
- 后端是否连接的是当前数据库。
- 浏览器请求是否带 Authorization。
- 登录接口是否 200。
- 登录后 `/auth/me` 是否 200。
- 是否有过期 token 触发全局登出。

### 普通成员看不到项目

可能原因：

- 没有加入项目。
- 项目不在“我参与的”条件里。
- 任务指派了但项目成员关系缺失。
- 项目处于归档或删除状态。

### 任务无法领取

可能原因：

- 任务不是可领取状态。
- 前置任务没完成。
- 前序环节任务为空。
- 成员没有必要标签。
- 翻译任务没有分配给该成员。
- 认领片段和别人重叠。
- 超过每人最大认领时长。

### 下一步任务没有启动

检查：

- 上一步任务是否审核通过。
- 翻译是否全部排序任务都通过。
- 后期是否拿到了最大翻译排序 ID 的结果。
- 前序环节是否有任务。
- 任务是否 frozen。
- 下游是否被级联重置。

### 文件上传失败

检查：

- 是否进入项目文件页。
- 是否选择了具体项目。
- 是否有 project ID。
- 上传策略是否允许该文件类型。
- 文件大小是否超过策略。
- S3 CORS 是否允许。
- S3 endpoint 浏览器是否可访问。
- 存储后端是否启用。
- 项目是否绑定存储后端。

### 头像或 logo 刷新后不显示

检查：

- 文件是否上传到默认存储后端。
- 存储后端是否可读。
- 后端头像代理接口是否正常。
- 旧头像是否已删除。
- 浏览器缓存是否仍在用旧链接。

### QQ 通知没有 @ 到人

检查：

- 项目是否填写 QQ 群。
- 成员是否绑定 QQ。
- QQ 桥接器是否在线。
- OneBot 是否连接。
- 机器人是否在群里。
- bot secret 是否一致。
- 系统设置 QQ 测试是否通过。

### 发布产物消失

如果上游任务被重置，发布任务会丢弃旧发布产物。

这是设计行为。

原因是旧种子、旧网盘链接或旧发布包不能继续流转。

### 同名文件没有合并

这是设计行为。

同名文件默认创建独立实体。

要追加版本，上传时必须选择“显式替换已有文件”。

### 翻译后期拿错上一流程文件

检查：

- 翻译排序 ID 是否正确。
- 上一排序 ID 是否审核通过。
- 后期是否使用最大排序 ID 的翻译结果。
- 是否有人重置了中间翻译任务。

## 推荐工作流模板

### 标准番剧单集流程

1. 监制创建项目并填写项目 QQ 群。
2. 监制确认分集数量。
3. 监制创建第 1 集片源整理任务。
4. 片源上传文件或链接并填写时长。
5. 监制审核片源。
6. 监制创建时轴制作任务。
7. 时轴下载片源并提交 ASS。
8. 监制审核时轴。
9. 监制按时长创建多个翻译任务并设置排序 ID。
10. 译者按顺序认领片段。
11. 第一个译者下载时轴结果并提交。
12. 监制审核第一个翻译任务。
13. 第二个译者下载第一个翻译通过结果并继续。
14. 重复直到最后一个翻译任务通过。
15. 监制创建后期任务。
16. 后期下载最终翻译结果并提交特效字幕和字体。
17. 监制审核后期。
18. 监制创建压制任务。
19. 压制按成品配置提交视频或网盘链接。
20. 监制审核压制。
21. 监制创建发布任务。
22. 发布提交种子、网盘链接、发布说明。
23. 监制审核发布。
24. 检查交付清单。
25. 发布项目公告或完成说明。
26. 项目全部集完成后归档。

### 快速小项目流程

适用于短片、PV、单集小项目：

1. 创建项目。
2. 创建片源任务。
3. 片源通过。
4. 创建时轴任务。
5. 时轴通过。
6. 创建一个翻译任务，排序 ID 为 1。
7. 翻译通过。
8. 如果不需要复杂后期，可创建压制任务前由监制确认是否跳过后期。
9. 压制通过。
10. 发布通过。

### 多人翻译流程建议

建议：

- 不要按人数平均乱切。
- 先按剧情段落或自然断点切。
- 每段长度不要超过模板或项目设置的每人最大时长。
- 每个翻译任务排序 ID 必须和段落顺序一致。
- 每段通过后再通知下一段开始。
- 术语统一放 Wiki，不要在聊天里散落。

### 发布前检查清单

发布前监制应确认：

- 片源通过。
- 时轴通过。
- 所有翻译排序任务通过。
- 后期通过。
- 压制通过。
- 文件版本使用的是最新审核通过版本。
- 成品配置符合模板。
- 字体包完整。
- 发布文件命名正确。
- 网盘链接有效。
- 种子或发布包可下载。
- 项目公告已写。
- QQ 群通知可达。

### 归档前检查清单

归档前确认：

- 所有集都完成。
- 发布任务通过。
- 文件版本完整。
- Wiki 和术语表保留。
- 项目公告留有最终说明。
- 关键评论已经处理。
- 没有未完成的冲突。
- 没有未审核的提交。
- 项目成员知道项目已归档。
