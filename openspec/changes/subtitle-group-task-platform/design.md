# Design

## Context

当前仓库只有一份高层规划文档和 OpenSpec 骨架，尚未形成可实施的产品规格。目标系统同时涉及项目模板、岗位工作流、文件上传与版本管理、ASS 字幕处理、通知投递、注册准入控制、项目 wiki 和项目级文件视图，属于跨前端、后端、存储、异步任务和数据模型的复合变更。

系统的关键约束有三类：

- 协作约束：片源、时轴、翻译、后期、压制、监制等岗位存在明确依赖关系，且既要支持指派，也要支持进组和接单。
- 结构约束：项目单元需要按季组织，而不是按单集组织，单集差异由季内任务、批次或元数据承载。
- 资产约束：同一项目会产生大量二进制文件和链接型交付物，必须提供统一项目文件筒、版本链和审核锁版，且不能静默覆盖旧版本。
- 处理约束：平台主要面向 ASS 字幕组，需要服务端理解 ASS 结构，支持多人分段翻译后的合并、时间轴去重和冲突复核。
- 准入约束：系统允许开放注册，但注册是否可用、是否要求群验证必须由管理员后台统一配置，并在待验证阶段阻止账号正常登录。

相关角色包括超级管理员、组管理员、监制和普通成员。外部依赖包括 S3 兼容对象存储、本地文件系统、SMTP 和 QQ NoneBot HTTP 接口，以及用于接收指定 QQ 群消息和发送群 @ 消息的 NoneBot 事件回调。

## Goals / Non-Goals

**Goals:**

- 定义一套以前后端分离为基础、以模块化单体后端为核心的实现方案。
- 将项目模板、季维度项目单元、工作流状态机、文件筒与版本链、字幕处理、通知投递、注册准入控制和项目 wiki 落成稳定的能力边界。
- 为后续 `/opsx:apply` 提供足够清晰的模块拆分、数据模型和集成策略。
- 确保版本管理、审核锁版、通知送达、注册验证和冲突处理都具备可审计性。

**Non-Goals:**

- 不在本次设计中定义视觉稿、像素级 UI 或前端样式系统。
- 不在本次设计中引入微服务拆分、跨组织多租户隔离或复杂即时聊天系统。
- 不在本次设计中落定具体认证提供商、邮件服务商或 QQ 机器人部署细节。
- 不在本次设计中实现 ASS 文本差异高亮编辑器，仅定义在线去重和冲突处理边界。

## Decisions

### 1. 使用前后端分离加模块化单体后端

后端采用模块化单体，按 Auth、Project、Template、Task、Workflow、File、Review、Notification、Subtitle Processing、Audit 等模块划分；前端采用单页应用，通过 REST API 访问后端。

这样做的原因是业务复杂度集中在规则编排和数据一致性，而不是服务规模。单体更容易保持事务边界、审计一致性和开发速度，同时保留未来拆出 File 或 Notification 服务的可能性。

备选方案：

- 微服务架构：过早引入服务拆分会增加部署、监控和一致性成本，当前阶段不成立。
- 全栈单仓同构页面：不利于文件直传、异步处理和清晰的系统边界。

### 2. 将项目模板作为按季工作流蓝图，而不是简单默认值集合

项目模板需要保存人员槽位、默认岗位链、任务依赖、上传策略、通知策略、ASS 处理策略和进组/接单规则。项目实例只填写标题、季维度项目单元、人员映射和必要排期信息。

这样做可以把“快速开项”真正落实为按季工作流实例化，而不是复制一堆可变表单。模板同时也是权限、上传和通知规则的继承入口。单集差异如果需要记录，放入季内任务、字幕批次或附加元数据，而不是把项目单元拆成按集结构。

备选方案：

- 仅复制标题和任务清单：无法稳定继承上传规则、通知规则和字幕处理策略。
- 把所有规则写死在代码里：后续调整番剧、电影、合集模板的成本过高。

### 3. 建立统一资产模型，项目文件筒与版本链覆盖二进制交付物

二进制文件交付物（视频、ASS、工程包、字体等）统一落入 `files` 和 `file_versions` 体系，写入对象存储或本地文件系统。任何"覆盖式"提交都创建新版本，维护当前版本、最新版本和最新审核通过版本指针。网盘链接类交付物使用独立模型（`link_history`），不走 `file_versions` 版本链。文件筒在展示层聚合两种资产。

项目文件筒基于聚合视图展示项目级全部资产，并对多版本文件提供明确标记和右键历史入口。

备选方案：

- 所有交付物统一落入 `file_versions`：网盘链接没有内容、大小和哈希，版本语义与二进制文件不同，硬塞会导致字段空置和语义扭曲。
- 直接覆盖原对象：会破坏审核追踪、回滚能力和下载审计。

### 4. 将 ASS 解析、字幕合并、版本对比和去重实现为后端异步处理能力

字幕处理模块负责解析 ASS 文件中的结构化信息，依据翻译任务的时间范围和来源版本生成合并作业。合并逻辑自动处理完全重复内容，对时间轴重合且文本冲突的条目生成冲突记录，并通过在线去重页交给人工复核。在线去重页从第一版开始就同时支持版本间对比、时间轴重叠可视化和冲突解决，而不是把版本对比留到后续版本。

这样做可以保证数据源一致、减少前端计算压力，并让冲突处理结果回到统一版本链和审计日志中。

备选方案：

- 在前端执行合并与去重：难以保证多人并发下的一致性，也不利于审计和重试。
- 只保存上传文件不做结构化解析：无法满足 ASS 定向优化和在线去重需求。

### 5. 注册采用管理员可配置的准入模式和群指令验证状态机

注册策略放在管理员后台的全局配置中，至少支持三种模式：禁止注册、允许注册但直接激活、允许注册但需要 QQ 群验证。处于群验证模式时，用户提交注册请求后创建账号和注册验证记录，账号状态设为 `pending_verification`，并生成一个随机八位、由英文和数字混合组成的校验码。

待验证账号在登录时不会进入正常系统，而是返回专用的待批准响应，其中包含管理员配置的 QQ 群号、当前账号对应的 `/verify <code>` 指令和可直接复制的展示字段。验证码在验证未通过前不设置过期时间，只在验证成功或被管理员显式重置时失效。NoneBot 监听指定 QQ 群消息后，将匹配到的 `/verify` 指令推送给后端；后端校验群号、验证码和账号状态后，将账号转为激活状态，并立即删除或清空验证码与账号的关联关系，防止再次复用。

这样做可以把公开注册和字幕组内部准入结合起来，避免任何拿到注册链接的人直接获得访问权限，同时不需要管理员逐个手工审核每个注册请求。

备选方案：

- 全部改为管理员手工创建账号：会明显提高运维成本，也不符合当前“允许注册”的需求。
- 使用邮件验证码代替 QQ 群验证：无法利用字幕组现有 QQ 群作为准入入口，也不满足指定群校验的要求。

### 6. 通知通过适配层投递到邮件与 QQ NoneBot，并保留送达日志和重试队列

任务、审核、进组审批、文件处理和字幕合并等事件统一进入通知模块，由通知模块解析收件人集合、应用用户偏好、生成消息体并分别调用 SMTP 和 NoneBot HTTP 接口。邮件按单收件人发送；QQ 渠道则向配置的项目群或工作群发送群消息，并使用 @ 的方式明确提醒对应人员。每一次投递都写入 `notification_deliveries`，失败进入重试队列。

这样做可以避免流程推进依赖“尽力而为”的消息发送，并为后续新增 Telegram、Discord 等渠道留下适配层接口。

备选方案：

- 业务模块直接发消息：会导致收件规则、失败处理和日志能力分散。
- 仅保留站内通知：无法满足当前对邮件和 QQ 的明确要求。

### 7. 项目 wiki 采用项目级文档加结构化表格的混合模型

项目 wiki 作为项目级知识面板存在，重点承载专有名词、命名规范、术语说明、翻译约定和制作备注。后端为 wiki 提供两种互补表达：Markdown 文档块和结构化表格块。前者适合写说明、规范和注释，后者适合维护术语对照、角色名、地名、注释规则等稳定数据。

这样做可以让字幕组在一个页面内同时维护自由文本和规范化词条，不必在纯 Markdown 和纯表单之间二选一。wiki 内容作为项目内共享知识，不应混入文件版本链，而是维护自己的文档版本或更新时间记录。

备选方案：

- 只支持 Markdown：术语维护和批量查阅成本偏高。
- 只支持结构化表格：说明类内容表达能力不足，无法覆盖制作约定和备注。

### 8. 数据核心使用关系型数据库，关键流程以显式状态和审计记录建模

项目、模板、季维度项目单元、任务、依赖、进组申请、审核、文件版本、翻译提交、冲突处理、通知送达、注册验证和项目 wiki 文档都有明确关联关系和事务边界，关系型数据库更适合支撑一致性和检索需求。关键状态变化必须分别写入任务状态日志、审核记录和审计日志，而不是仅依赖主表当前值。

备选方案：

- 文档型数据库：不利于复杂筛选、关联校验和事务性回滚。
- 仅依赖事件流重放：当前阶段会提高实现复杂度，不适合首期落地。

### 9. 翻译稿直接进入文件版本链，合并结果作为独立文件实体

翻译者上传的 ASS 翻译稿在提交时即作为独立的文件实体进入统一的 `file_versions` 版本链，而不是先作为游离的翻译提交记录存在。多人翻译稿各自形成独立的文件实体链，服务端合并结果生成一个新的"合并实体"，拥有自己独立的版本链。原始稿实体与合并实体之间通过 `translation_submissions` 和 `merge_jobs` 关联表建立可追溯的链接。

这样做可以保证文件筒能统一展示翻译稿和合并稿的完整演进链路，同时保留分段翻译的元数据（覆盖时间段、来源任务、上传人）。合并稿独立成实体避免了翻译稿版本与合并结果版本在同一条链上互相干扰。

备选方案：

- 翻译稿不入版本链，合并后再入链：会导致文件筒无法展示原始翻译稿，失去来源追溯能力。
- 原始稿和合并稿在同一条版本链上：合并稿成为某个翻译稿实体的 vN，语义混乱，难以区分"人工上传"和"系统自动生成"。

### 10. 文件实体以每次上传为单位独立创建，同名不自动合并

无论文件名是否相同，每次上传操作都创建一个独立的文件实体。用户如需"替换已有文件"，必须在上传界面显式选择目标实体。文件筒中同名文件通过来源任务、上传人、上传时间区分。

这样做避免了"同任务同名 = 同一文件"的隐含假设导致的误覆盖，特别是在片源岗位多次上传同名原始视频文件或翻译者上传同名 ASS 文件时，能确保每份上传都被完整保留，直到用户显式确认替换意图。

备选方案：

- 按任务+文件名自动去重并追加版本：容易在不知情的情况下覆盖他人上传的文件。
- 按内容哈希自动去重：计算开销大，且哈希碰撞和故意覆盖场景难以处理。

### 11. 网盘链接类交付物使用独立历史记录表，不走统一文件版本链

网盘链接（百度盘、Google Drive、115 等）不进入二进制文件统一的 `file_versions` 体系，而是在 `link_history` 表中独立维护变更轨迹，记录链接地址、提取码、有效期、来源说明和校验状态的每次变更。文件筒查询时合并展示二进制文件实体和链接型交付物，但底层模型分离。

这样做是因为网盘链接没有文件内容、大小和哈希，"版本"语义与二进制文件不同。硬塞进统一版本链会导致字段空置和语义扭曲。

备选方案：

- 网盘链接也走 `file_versions`：大量字段空置，版本号语义不一致。
- 网盘链接只更新当前记录不保留历史：丢失了链接变更的审计能力。

### 12. 文件版本指针采用自动推算语义

每个文件实体维护三个版本指针：

- `当前版本`：自动推算，优先指向"最新审核通过版本"，若无则指向"最新版本"。用户不可手动切换，简化文件筒交互并确保默认拿到的是安全版本。
- `最新版本`：自动跟随每次上传，指向上传时间最晚的版本。
- `最新审核通过版本`：审核通过时自动更新，用于下游任务解锁和安全回退。

备选方案：

- 允许手动切换当前版本：增加 UI 复杂度和权限管理成本，且容易因选错版本导致下游使用未审核内容。

### 13. 审核驳回时通过独立快照表保留审核时刻状态

审核驳回时不在文件版本链中创建新快照版本，而是在 `review_snapshots` 表中写入审核记录，引用被审核时的文件版本号、哈希和元数据。驳回意见与快照引用绑定，便于后续多轮审核回溯。

这样做轻量且不会膨胀版本链，同时保留了"这一刻审的是哪个版本"的显式记录。

备选方案：

- 驳回时创建新版本快照：快速膨胀版本链，且"快照版本"与正常版本语义混杂。
- 只记录版本号不存快照表：丢失了审核时刻的完整元数据上下文。

### 14. 任务取消只冻结未开始的下游任务，成员可主动退回被指派的任务

任务被取消时，处于"可领取"或"已指派"但尚未"进行中"的下游任务自动回退到冻结状态；已经"进行中"或更后的下游任务不受影响，但系统向监制推送警告提醒。被指派的成员可以主动将任务退回至"可领取"状态，不需要审批，但会记录审计日志。

这样做的原因是避免上游取消导致大量已在进行的工作被撤销，同时给予成员适当的自主权处理不合适的指派。

备选方案：

- 级联冻结所有下游任务：过于严格，已开始的下游工作会被强制中断。
- 不允许退回被指派的任务：实际操作中会很僵，成员只能找监制私下协调。

### 15. 在线去重页面仅对监制和指定角色开放编辑权限

在线去重页面的冲突处理操作（确认保留项、合并项、标记待处理）仅允许监制或被项目指定的冲突处理人员执行，普通翻译成员只能查看冲突可视化。单人编辑自然避免了并发冲突问题，无需引入乐观锁或悲观锁机制。

备选方案：

- 开放给所有成员编辑：必须引入复杂的并发控制（乐观锁/悲观锁），且冲突处理责任分散。

### 16. 组管理员与监制权限按场景分离

组管理员负责成员管理、标签、公告和项目模板维护，日常不干预具体项目。当监制离职或长期缺席时，组管理员可以"接管"项目成为临时监制，接管行为写入审计日志，原监制回来后可以归还权限。

这样做避免了日常管理中组管理员和监制的权限重叠，同时在异常情况下提供了兜底的权限转移能力。

备选方案：

- 组管理员拥有所有项目完全权限：削弱监制的项目自主权，容易造成误操作。
- 完全分离，不支持接管：监制缺席时项目陷入无人管理状态。

### 17. 模板修改只影响新创建的项目

项目模板的上传策略、通知策略、ASS 处理策略、岗位链等规则被修改后，只影响此后基于该模板创建的新项目。已有项目保持创建时继承的规则不变，避免进行中的项目因模板突变而行为异常。

备选方案：

- 全部追溯：现有项目的规则突然变化，可能导致进行中的任务流程中断。
- 可选追溯：增加管理员的决策负担和系统的交互复杂度。

### 18. 项目归档后只读但允许复盘评论，支持取消归档恢复

项目归档后，任务和文件进入只读状态，不允许修改和上传新文件，但允许成员发表评论（用于复盘总结）和上传复盘相关文件。评论通知正常发送。归档项目对普通成员默认不可见，只有管理员和项目相关人员可以查看。监制或管理员可以随时"取消归档"将项目恢复为活跃状态。

备选方案：

- 完全只读：丧失了复盘讨论的能力。
- 归档后不可恢复：一旦归档就无法继续编辑，灵活性不足。

### 19. 项目删除采用"先归档、后软删除、再回收站清理"的三阶段策略

项目必须先归档才能触发删除。删除操作为软删除（标记 `deleted` 状态），项目进入回收站。回收站中的项目保留天数由管理员后台配置，超期后由后台任务执行物理清理。从回收站恢复的项目回到归档状态。

归档项目的旧版本清理规则：归档满 N 天（管理员配置）后自动清理旧版本，只保留每个流程的定稿/通过版本。若归档后 N 天内触发删除，则跳过版本清理（项目整体将被删除）；若从回收站恢复，恢复后继续正常执行版本清理流程。

备选方案：

- 直接物理删除：无法恢复误删项目，风险过高。
- 不删除只归档：数据无限增长，存储成本不可控。

### 20. 搜索按模块分别实现，不引入全文搜索引擎

项目/任务列表支持按名称的 LIKE 模糊查询；文件筒支持按文件名和标签过滤；评论内容首期不纳入搜索范围。不引入 Elasticsearch 等全文搜索引擎，降低部署复杂度。

备选方案：

- 全文搜索引擎：部署和运维成本高，首期搜索量不足以 justify。
- 完全不做搜索：无法满足"项目分散找不到"的核心痛点。

### 21. 通知采用渠道升级策略

同一条通知的投递按渠道优先级逐级升级：先发站内通知，N 小时未读后发送邮件，再 N 小时未读后发送 QQ 群消息。升级时间间隔和是否启用升级由管理员后台配置。

备选方案：

- 所有渠道同时发送：可能造成刷屏和过度打扰。
- 仅保留站内通知：无法满足邮件和 QQ 的明确要求。

### 22. 任务评论默认静默，仅 @ 时触发通知

任务评论区的普通评论不触发任何通知，避免刷屏。评论中 @ 提到的成员会收到定向通知。任务状态变更（指派、领取、审核通过、驳回等）继续按原有规则向相关人员发送通知。

备选方案：

- 每条评论都通知全员：评论量大的任务会产生严重的通知噪音。
- 评论不做任何通知：评论区沦为留言板，响应性极差。

### 23. 资格标签采用注册申请 + 管理员审批模式

用户注册时可以在技能档案中自选意向资格标签（如压制、后期、翻译）。服务端维护用户粒度的标签状态列表，记录每个标签处于"申请中"还是"已授予"。管理员对申请者进行考核后手动授予或拒绝标签。

备选方案：

- 完成任务自动获得：无法保证质量，新手误操作后自动获得高级标签。
- 管理员直接分配，不支持申请：增加管理员主动发现人才的成本。

### 24. 最终交付清单由模板继承、项目级可编辑

项目模板中定义该类型的默认交付清单（如番剧需要成品视频、外挂字幕、字体包、封面等）。开项时自动继承模板的交付清单。项目创建后，监制可以在项目设置中覆盖或微调清单内容。最终结项时的清单校验以项目级配置为准。

备选方案：

- 硬编码清单：不支持不同项目类型的差异（番剧 vs 电影 vs 合集）。
- 完全项目级自由编辑：每个项目都从零配置，效率低下。

### 25. 任务改派时向原被指派人发送取消通知

任务从指派人 A 改派给 B 时，A 会收到一条"任务已转交，你不再负责该任务"的通知，B 收到正常的"新任务指派"通知。双方都在通知中心看到明确的权责变化记录。

备选方案：

- 不处理旧通知：A 可能误以为仍在负责该任务。
- 原通知自动更新：通知通常是 immutable 的，技术上难以实现动态修改。

### 26. 空槽位由监制手动指派或配置为开放领取

项目模板定义的人员槽位在开项时未填满的情况下，对应岗位的任务状态为"待发布"，监制可以在项目页面手动指定人员，或将该槽位配置为"开放领取"。空槽位不会自动生成"可领取"任务，避免无人负责的任务被误接。

备选方案：

- 空槽位自动开放领取：可能导致关键岗位任务被不合格人员接取。
- 开项强制填满所有槽位：操作过于严格，实际开项时人员往往尚未全部确定。

### 27. 翻译采用竞争式自选区间领取

整集翻译不预先拆分为固定子任务。片源上传时填写集长度，翻译任务开放后，翻译者自行申报想要翻译的时间区间。监制在项目模板中配置每位翻译最大可接取的片段长度（模板默认值，项目期间可热更改）。翻译者申报的区间经系统校验无重叠且未超过个人限额后，生成对应的翻译子任务并分配给该翻译者。所有片段被领完后锁定领取，除非有人放弃释放区间。

备选方案：

- 模板预拆分固定区间：灵活性差，不同集的片段长度需求不同。
- 监制手动拆分：工作量大，且拆分时机往往在开项后很久。

### 28. 去掉任务优先级字段

除翻译岗位支持多人并行分段处理外，片源、时轴、后期、压制、监制等岗位严格串行执行，形成单向流水线。任务之间不存在并发竞争，不需要优先级字段。列表排序按截止时间或创建时间即可。

备选方案：

- 保留优先级字段：串行流程中优先级无实际意义，增加不必要的复杂度。

### 29. 超期任务自动标记但不做自动回收

任务超期后系统自动将状态标记为"超期"，在看板中以红色高亮显示，并向监制和管理员发送升级通知。系统不做任何自动回收、改派或取消操作，由监制人工处理（催办、改派、延期等）。

备选方案：

- 超期自动回收：字幕组为志愿性质，自动回收过于强硬，容易引发成员不满。
- 超期不做任何处理：任务可能永久挂死在看板中，监制难以发现。

### 30. 项目动态时间线分项目级和个人级

每个项目详情页提供"动态"Tab，按时间倒序展示该项目下的全部事件（任务状态变更、文件上传、成员进出、审核记录等）。同时提供全局个人首页，聚合展示当前用户参与的所有项目的最新动态。两种视图互补，分别满足项目聚焦和个人待办的需求。

备选方案：

- 只做项目级时间线：成员参与多个项目时，需要逐个进入项目查看动态，效率低。
- 只做全局时间线：缺少项目上下文，难以聚焦单项目的完整演变。

### 31. 公告分全局公告和项目公告

全局公告由组管理员发布，所有登录用户可见（如停服通知、招募信息）。项目公告由监制或组管理员在项目内发布，仅该项目成员可见（如项目进度说明、注意事项）。两套公告独立管理、独立展示。

备选方案：

- 只做全局公告：项目内的重要通知缺乏针对性。
- 只做项目公告：系统级通知无法触达所有人。

### 32. 批量操作只做关键批量

首期仅支持两类批量操作：一是批量指派（将一个项目单元下的多个任务一次性指派给同一人），二是批量归档（结项时一次性归档整个项目的所有单元）。其他操作（批量审核、批量删除等）暂不支持，避免交互复杂度和误操作风险。

备选方案：

- 全部批量操作：实现成本高，且字幕组日常操作中批量场景有限。
- 不做任何批量：一个 12 集项目有 72 个任务，逐个操作过于痛苦。

### 33. 流水线关键节点增加监制审核关卡

每集的生产流水线在翻译、后期、压制和发布四个阶段的提交后，都必须经过监制审核通过才能解锁下一阶段。审核不通过则打回修改。片源和时轴阶段不需要单独审核（片源上传即完成，时轴完成后自然进入翻译阶段）。发布为新增阶段，负责最终成品的对外发布确认。

备选方案：

- 只有最终审核：中间阶段缺乏质量控制，问题可能累积到后期才发现。
- 每个岗位都审核：过于繁琐，片源和时轴的输出相对标准化，不需要每步都审。

### 34. 文件下载采用临时链接 + 敏感标签控制

文件下载通过临时链接实现：S3 存储使用预签名 URL，本地存储使用数据库记录的临时链接，超时后自动销毁。链接有效时间由管理员全局配置，项目上监制可覆盖。默认项目成员均可下载，但文件可标记"敏感"标签，标记后仅特定角色（如片源、压制、监制）可下载。

备选方案：

- 永久链接 + 每次请求鉴权：实现简单但裸链暴露风险高。
- 纯时效控制不做权限区分：无法对片源等敏感文件做额外保护。

### 35. 成品配置由模板定义、项目级可覆盖

压制任务的成品配置（分辨率、码率、编码器、封装格式、命名规则等）在项目模板中预设，开项时自动继承。项目创建后，监制可以在项目设置中覆盖或微调配置。最终压制任务按项目级配置执行。

备选方案：

- 压制者自行配置：缺乏统一规范，同一项目的不同集可能出现规格不一致。
- 完全模板固定：特殊情况（如某项目需要 4K）无法灵活处理。

### 36. 任务评论不上传新文件，支持行级评论

任务评论区禁止上传新文件附件。评论可以引用任务下已存在的某个文件版本，对文件整体发表意见。在翻译/字幕阶段的任务评论区，额外支持选择 ASS 文件的某一行进行行级评论（类似代码审查）。行级评论绑定到具体文件版本和行号。

备选方案：

- 允许上传附件：评论区附件量大但价值低，增加存储和版本链负担。
- 不支持行级评论：字幕逐行校对时，无法精确定位问题所在。

### 37. 项目标签仅用于分类筛选

项目的标签属性仅用于项目列表的分类和筛选（如"番剧"、"电影"、"急"等），不参与任何自动化规则、模板选择或优先级计算。标签由组管理员维护标签库，创建项目时选择。

备选方案：

- 标签触发规则：增加系统的隐式复杂度，标签变更可能导致意外行为。
- 标签关联模板：标签和模板的映射关系需要额外维护，开项流程变复杂。

### 38. 工作负载视图分个人/项目/全局三级

个人看板展示当前用户自己接取或被分配的全部任务。监制看板展示其负责的项目下的所有成员及对应任务状态。管理员全局看板展示所有成员在所有项目中的任务分布。三级视图权限范围层层递进。

备选方案：

- 统一看板：权限控制复杂，普通成员不应看到其他人的任务详情。
- 只做个人看板：管理员和监制缺乏统筹视角。

### 39. Wiki 不保留编辑历史，但支持审批流配置

项目 wiki 每次保存直接覆盖当前内容，只记录最后更新时间和更新人，不保留历史版本。通过审计日志可追溯到谁改了 wiki，但无法查看改了什么或回滚。Wiki 编辑支持审批流：管理员可在后台全局配置是否需要 wiki 审批，项目上监制可手动覆盖该设置。开启审批后，wiki 修改需经监制审批通过才生效。

备选方案：

- 保留编辑历史：wiki 修改频率低，保留历史的收益不高，但增加存储和查询成本。
- 强制审批：wiki 内容通常为非关键协作说明，强制审批会降低更新积极性。

### 40. 集长度存储在项目单元，翻译区间领取使用独立记录表

`project_units` 表增加 `episode_length` 字段，由片源上传者在提交片源时填写。翻译任务开放后，翻译者在项目单元粒度上自选时间区间申报领取。系统通过 `translation_claims` 表记录每次领取（外键指向 `project_unit_id`），包含申报人、起止时间、状态（claimed / released / completed）。领取时校验区间不重叠、个人未超限额，全部片段被领完后锁定领取通道。

备选方案：

- 集长度存在任务表：片源任务完成后的集长度需要跨任务传递，增加耦合。
- 不存领取记录，直接拆分为固定子任务：失去竞争式领取的灵活性。

### 41. Wiki 审批流使用单表状态字段 + pending_content 存储待审内容

`wiki_documents` 表增加 `status` 字段（draft / pending / approved）和 `pending_content` 字段。前端显示内容由所有 `approved` 状态的记录拼装而成。当用户提交修改且审批流开启时，内容存入 `pending_content` 并将状态设为 `pending`。监制或系统管理员在审核页面以 diff 形式查看待审内容，审批通过后覆盖正式内容并更新状态为 `approved`。每个 pending 内容视为一个独立的 wiki Patch。

备选方案：

- 独立待审表：增加表数量，但 wiki 审批频率低，单表字段足够承载。
- 不存储待审内容，仅记录 diff 摘要：审核人无法看到完整修改上下文。

### 42. 任务评论使用统一表，通过可空字段区分普通评论、文件引用和行级评论

`comments` 表统一承载所有任务评论。普通评论的 `file_version_id` 和 `line_number` 均为空；引用某个文件版本发表评论时填写 `file_version_id`；在翻译/字幕任务中对 ASS 某一行发表评论时同时填写 `file_version_id` 和 `line_number`。不拆分为多个评论表，避免评论查询时的联合操作和排序复杂度。

备选方案：

- 拆分为 `comments` 和 `line_comments` 两个表：查询时需要 UNION，排序和分页复杂。
- 行级评论纯文本标注：无法精确关联到文件版本和行号，回查困难。

### 43. 下载链接由有权限的用户点击自动生成，复用未失效链接，临期自动换新

任何拥有文件查看权限的用户点击下载按钮时，系统自动生成临时下载链接。若该用户对该文件已存在未过期的链接，则直接复用。若现有链接距失效时间不足 30 秒（视频文件为 60 秒），则视为已失效并创建新链接。链接有效期由管理员全局配置、项目上监制可覆盖，但系统强制下限为 90 秒，不允许配置更短的有效期。过期链接由每 30 秒运行的后台任务扫描清理。

备选方案：

- 每次点击都生成新链接：数据库快速膨胀，链接记录大量冗余。
- 永久链接 + 每次鉴权：裸链暴露风险高，且无法支持预签名 URL 等临时凭证机制。

### 44. 发布为独立岗位，支持任务类型配置和前置修改自动重置

发布是流水线中的独立岗位（与片源、时轴、翻译、后期、压制并列），模板中配置"发布"人员槽位。发布任务类型可在模板中配置：种子 / 种子+网盘 / 网盘 / 其他渠道。发布岗负责上传种子文件、网盘链接或其他发布所需内容。当发布任务的前置任务（编码或更早阶段）发生修改后，发布任务自动重置为 pending 状态，并丢弃之前上传的发布产物，允许发布岗重新上传新的发布内容。

备选方案：

- 发布作为监制的子职责：增加监制工作负担，且发布操作（制作种子、上传网盘）通常由专门人员负责。
- 发布完全手动触发：前置修改后已发布的旧内容可能与新版本不一致，缺乏自动兜底。

### 45. 前置任务修改或重置时，下游已完成任务自动级联重置，非发布岗位保留文件历史

流水线中任意岗位的任务被修改或被人为重置为"进行中"状态时，系统自动将该任务下游所有已提交或已完成状态的岗位任务重置为"进行中"（或对应前置状态），以提示下游成员前置内容已变更需要重新处理。非发布岗位（片源、时轴、翻译、后期、压制）重置时保留该岗位此前上传的全部文件版本历史，任务回到待处理/进行中状态后成员可基于历史文件继续工作。发布岗位（Decision #44）重置时丢弃已上传的发布产物（种子、网盘链接等），因为发布产物与前置内容强绑定，旧产物不可复用。

级联重置的触发边界：

- 片源修改 → 时轴、翻译、后期、压制、发布全部重置
- 时轴修改 → 翻译、后期、压制、发布重置
- 翻译修改 → 后期、压制、发布重置
- 后期修改 → 压制、发布重置
- 压制修改 → 发布重置

系统向被重置任务的负责人发送通知，说明因前置任务变更导致任务需要重新处理。

备选方案：

- 不自动重置下游：需要人工逐个检查，容易遗漏已失效的下游产出。
- 所有岗位重置时都丢弃文件：翻译稿、时轴文件等可能仍可复用，全部丢弃代价过高。
- 所有岗位重置时都保留文件：发布产物（种子、网盘链接）与特定版本内容强绑定，保留旧产物会导致发布错误内容。

### 46. 用户资料支持头像和昵称

`users` 表增加 `nickname` 和 `avatar_url` 字段。每个用户拥有独立的资料页面，可修改昵称和上传头像。头像通过统一的存储适配层上传，文件路由到系统配置的默认存储后端。头像文件路径按用户维度隔离，避免不同用户的头像互相覆盖。

备选方案：

- 不区分昵称和用户名：字幕组成员通常使用圈名，需要独立于登录用户名的显示名称。
- 头像走独立存储服务：增加部署复杂度，复用已有存储适配层更合理。

### 47. 存储后端支持多实例化和空间配额管理

系统支持配置多个存储后端实例，每个实例可以是 S3 兼容对象存储或本地文件系统目录。`storage_backends` 表记录每个实例的配置（类型、地址、凭据、桶名/根目录、最大可用空间配额、当前已用空间）。管理员后台提供存储后端的增删改查和配额配置。多个实例并存时，系统按项目维度路由文件读写请求。

备选方案：

- 单存储后端：无法横向扩展，大项目或高峰期容易出现容量瓶颈。
- 按文件类型拆分后端：增加路由复杂度，且项目文件通常需要统一管理和备份。

### 48. 项目创建时由监制选择绑定的存储后端

`projects` 表增加 `storage_backend_id` 外键。项目创建（从模板实例化或手动新建）时，监制必须从管理员已配置的存储后端列表中选择一个作为该项目的文件存储位置。项目创建后，该项目下的所有文件上传（包括片源、字幕、工程文件等）均路由到绑定的存储后端。头像等用户级文件不走项目绑定后端，而是使用系统默认后端。

备选方案：

- 项目自动分配后端：需要复杂的负载均衡和容量预测算法，首期成本过高。
- 模板中预设后端：模板是通用配置，不应绑定到具体的存储实例。

### 49. 所有用户输入执行统一安全校验，防止注入与遍历攻击

系统对所有用户输入执行严格的安全校验作为横切基线要求：

- **SQL 注入防护**：所有数据库操作必须通过参数化查询或 ORM 完成，禁止任何字符串拼接 SQL 语句的场景。搜索接口的 LIKE 模糊查询使用参数化占位符。
- **XSS 防护**：所有输出到前端的用户生成内容（评论、公告、wiki、任务描述、昵称等）必须经过 HTML 转义。富文本输入（如 wiki Markdown）在渲染时做白名单过滤，禁止执行脚本。
- **文件上传安全**：上传文件校验 MIME 类型、扩展名白名单、大小上限。文件名去除路径遍历字符（`..`、`/`、`\` 等），存储时使用随机生成的内部标识符，不直接使用原始文件名作为存储路径。禁止上传可执行文件（`.exe`、`.sh`、`.php` 等）。
- **接口参数校验**：所有 REST API 入参执行长度限制、类型校验和范围校验。分页参数限制最大页大小，ID 参数校验格式，字符串参数限制最大长度。
- **路径遍历防护**：本地存储的文件读写操作使用规范化路径，确保文件始终落在配置的根目录内，禁止通过相对路径访问根目录外的文件。

备选方案：

- 每个模块自行实现安全校验：容易遗漏，标准不统一。
- 完全依赖 WAF 等外部防护：无法覆盖应用层的业务逻辑漏洞（如文件上传绕过）。

### 50. 前端技术栈和视觉风格以仓库中 `frontend/app` 模板为准

前端实现必须基于仓库 `frontend/app` 目录中已有的模板项目。模板已配置 React 19 + TypeScript + Vite + Tailwind CSS v3 + shadcn/ui（New York 风格，slate 基础色），并预置 40+ 个 UI 组件（Button、Card、Dialog、Form、Table 等）。状态管理使用 Zustand，路由使用 React Router v7，表单校验使用 React Hook Form + Zod，图标使用 Lucide React。

所有新增页面、组件和交互必须以该模板的组件库、配色方案、布局模式和代码组织方式为基准，禁止引入与模板风格冲突的第三方 UI 库或自定义样式系统。新增页面复用模板中已有的页面骨架（AppShell、路由结构、状态管理 store 模式）。前端任务实现时从模板出发，在其上扩展业务功能，而不是重写基础设施。

备选方案：

- 允许各前端任务自行选择 UI 库：会导致页面风格不统一，维护成本陡增。
- 重新搭建前端框架：已有模板完成了基础设施配置，重复搭建是浪费。

## Risks / Trade-offs

- [工作流规则过多导致实现膨胀] → 以项目模板、岗位链和显式状态机为核心，只覆盖当前六类岗位和必要流转。
- [项目文件筒查询压力较大] → 用项目、任务、文件类型、更新时间等字段建立索引，并将文件内容与元数据分离存储。
- [ASS 合并规则复杂且存在边界案例] → 从第一版开始支持结构解析、版本对比、时间轴重叠可视化和冲突分组，但仍把复杂编辑留给人工确认。
- [群验证指令可能被伪造或发送到错误群] → 只接受管理员配置的目标群号事件，并校验 NoneBot 回调签名、验证码状态和账号绑定关系。
- [QQ群 @ 通知可能产生刷屏和误提醒] → 在项目级配置群号和提醒粒度，并记录群消息投递对象与消息体，便于审计和调优。
- [QQ NoneBot 接口契约不稳定] → 通过通知适配层封装，并将失败投递、重试和人工补发纳入送达日志。
- [版本链和审核锁版容易互相影响] → 明确区分当前版本、最新版本和最新审核通过版本三类指针，所有状态变更写审计日志。

## Migration Plan

1. 在空仓库中先创建 OpenSpec 规格，冻结能力边界。
2. 实现数据库基础模型和后端模块骨架，优先打通按季项目模板、项目实例化、任务流转与审计。
3. 实现注册策略配置、待验证账号状态机和 QQ 群验证回调链路。
4. 实现统一文件服务和版本链，再接入本地存储与 S3 适配。
5. 接入通知模块和外部渠道，补齐群 @ 消息、送达日志与重试机制。
6. 实现字幕处理模块、在线去重页面、版本间对比和冲突写回流程。
7. 实现项目 wiki 页面与 Markdown / 表格双模式编辑。
8. 上线时先以本地存储和测试通知通道验证端到端流程，再逐步切换正式对象存储和正式通知渠道。

回滚策略：

- 数据库迁移按模块分步发布，失败时回滚至前一版 schema。
- 外部渠道接入失败时保留站内通知和日志，不阻塞主业务数据写入。
- 注册群验证链路异常时保留待验证状态，不提前激活账号，也不提前销毁验证码关联。
- 字幕合并失败时保留原始提交和冲突作业记录，不覆盖当前有效版本。
